## §1 — Wire-Spec

The plugin↔host MCP-call protocol uses **two DOM CustomEvents** that traverse
the host's shadow-DOM boundaries (`bubbles: true, composed: true`) to bridge
plugin custom-element bundles to the host's IPC layer.

### §1.1 Event-names (canonical)

```ts
const PLUGIN_MCP_CALL_EVENT = 'plugin:mcp-call' as const
const PLUGIN_MCP_RESPONSE_EVENT = 'plugin:mcp-response' as const
```

Both events MUST be dispatched with `bubbles: true, composed: true` to cross
custom-element + shadow-DOM boundaries. The `composed: true` flag is what
enables shadow-DOM traversal — without it, listeners outside the shadow tree
never see the event.

### §1.2 Request detail-shape (`plugin:mcp-call`)

```ts
interface PluginMcpCallDetail {
  request_id: string                    // RFC 4122 UUIDv4, generated via crypto.randomUUID
  qualified_name: string                // e.g. "wiz-mind.character.list"
  arguments: unknown                    // tool-specific args, forwarded verbatim to bridge
  actor_class?: 'user' | 'kiara'        // v0.6.1+ — optional, defaults to host policy
}
```

**`request_id`** is generated by Foundation `callMcp()` via `crypto.randomUUID()`.
On platforms missing `crypto.randomUUID` (insecure context — neither `https`
nor `localhost`), Foundation rejects with `CallMcpError('crypto_unavailable')`
before any event is dispatched.

**`qualified_name`** MUST follow the pattern `<plugin-id>.<verb>[.<sub-verb>]*`.
Cross-plugin call-attempts (e.g. `mind-canva.layout.create` from a wiz-mind
mount) are rejected host-side — see §6 Security Model.

**`actor_class`** (v0.6.1+) is propagated as the request's authorization-context.
Foundation **omits the field from the dispatched detail when undefined** —
matching the host-side parser's spread-omit pattern. Set to `'kiara'` for
autonomous-agent calls, `'user'` for explicit user-mirroring.

### §1.3 Response detail-shape (`plugin:mcp-response`)

```ts
type PluginMcpResponseDetail<T = unknown> =
  | { request_id: string; ok: true;  result: T }
  | { request_id: string; ok: false; code: string; message?: string }
```

Discriminated by `ok`. The `request_id` MUST match the corresponding
`plugin:mcp-call`'s `request_id` — Foundation's listener filters by this
field to support multiple in-flight calls sharing the same DOM listener.

Error-codes follow **Drift #103 canonical error-shape**. The host MAY emit
any string code; Foundation surfaces them via `CallMcpError.code`.

Foundation-emitted error-codes (NOT host-side):
- `crypto_unavailable` — `crypto.randomUUID` missing
- `timeout` — `options.timeoutMs` elapsed without matching response (v0.6.1+)

### §1.4 Dispatch-target invariant

> **The host MUST dispatch the `plugin:mcp-response` event on the same
> element-tree that dispatched the `plugin:mcp-call`** — specifically, on
> `event.target` of the request-event (or any ancestor — descendants will
> not receive bubbled events).

This invariant was discovered during the joint-smoke and is documented in
detail in §5 (DOM-Bubble-Direction Anekdote). A `dispatchEvent` on the wrong
node produces a **silent 30-second timeout** with no diagnostic — by far
the worst failure-mode discovered to date. Strict adherence is required.

### §1.5 Cross-plugin attack guard (host-enforced)

> The host MUST validate that `qualified_name.startsWith(activePluginId + '.')`.
> Prefix-attacks (e.g. `wiz-mindfoo.tool` from a `wiz-mind` mount) MUST be
> rejected with `code: 'forbidden_namespace'`.

Foundation **does not** enforce this — Foundation cannot know which plugin
is currently mounted; only the host has that context. See §6 Security Model
for the full enforcement-contract.

## §2 — Host-Side Responsibilities

> _Host-side implementation references in this section come from
> Theseus-Agent commits `e84b5d3` (Layer 7 plugin:mcp-call bridge),
> `0aff0c5` (DOM-bubble-direction fix), and `003d37f` (namespace-
> validation with 7 unit tests). Credit: agent (Luma)._

A host implementing the wire-spec from §1 must satisfy four
contracts. None of these are optional — Foundation's `callMcp()`
relies on every one of them to resolve correctly.

### §2.1 Listen on a stable ancestor of plugin mount-points

Plugin custom-elements dispatch `plugin:mcp-call` with
`bubbles: true, composed: true`. The host attaches its handler to
**any DOM-ancestor** of every active plugin mount-point — typically
the wrapper `<div>` the plugin HTML is injected into.

```ts
// Reference: Theseus-Agent apps/mymind/src/renderer-main/ChatApp.svelte
// in attachPluginEventListeners(mount: HTMLElement):
mount.addEventListener('plugin:mcp-call', onMcpCall)
```

Mount-binding (not document-binding) keeps the handler scoped to
the current plugin lifecycle. When the user switches plugins, the
handler tears down with the mount.

### §2.2 Validate namespace before dispatch

The `qualified_name` field MUST be prefixed by the dispatching
plugin's `pluginId` followed by `.`. This is the cross-plugin
attack defence: plugin A cannot trick the host into calling
plugin B's tools by spoofing the `qualified_name`.

The validation is a **pure prefix check** with two invariants:
- `startsWith('<pluginId>.')` accepts arbitrary dot-segment counts
  (`wiz-mind.scene.current` and `wiz-mind.x` both pass for
  `pluginId='wiz-mind'`)
- Bare-prefix attacks fail (`wiz-mindfoo.tool` does NOT pass for
  `pluginId='wiz-mind'` because the `.` separator is required)

```ts
// Reference: plugins-page-helpers.ts validateMcpCallNamespace
export type McpCallValidation =
  | 'ok' | 'no_active_plugin' | 'invalid_request' | 'forbidden_namespace'

export function validateMcpCallNamespace(
  activePluginId: string | null,
  qualifiedName: unknown,
): McpCallValidation {
  if (activePluginId === null || activePluginId.length === 0) return 'no_active_plugin'
  if (typeof qualifiedName !== 'string' || qualifiedName.length === 0) return 'invalid_request'
  return qualifiedName.startsWith(activePluginId + '.') ? 'ok' : 'forbidden_namespace'
}
```

Hosts SHOULD extract this into a pure, unit-tested function rather
than inlining the check — the prefix-attack edge case is easy to
miss in template code.

### §2.3 Route via the host's existing IPC + bridge layer

After validation passes, the host invokes its plugin-bridge
infrastructure. Tokens, JWTs, scope-checks, audit-logging — all
the security-sensitive plumbing — stay **main-process / server-side**.
The plugin bundle never sees a token.

```ts
// Reference: Theseus-Agent IPC pluginsDispatchTool handler signature:
const result = await api.pluginsDispatchTool({
  qualifiedName,             // e.g. 'wiz-mind.character.list'
  arguments: args,           // free-form JSON
  ...(actorClass !== undefined ? { actorClass } : {}),
})
// result: { ok: true, result } | { ok: false, code, message }
```

This is where capability/scope-checks happen. The host's bridge
verifies that the active plugin has the manifest-declared scope
for the requested tool before forwarding to the plugin-bridge HTTP
endpoint.

### §2.4 Dispatch the response on the request's ORIGIN target

The single most critical host-side invariant — the one that
caused the joint-smoke's silent 30s timeout (§5.0 / §5.1 below):

```ts
// CORRECT — fires Foundation's listener in target phase:
const responseTarget: EventTarget = event.target ?? mount
responseTarget.dispatchEvent(new CustomEvent('plugin:mcp-response', {
  detail: payload,
  bubbles: true,
  composed: true,
}))
```

`event.target` is the plugin custom-element where Foundation
registered its response-listener. Bubbling flows up (target →
ancestors), so dispatching on the parent never visits the child.

The `?? mount` fallback handles the (theoretical) case where
`event.target` is null — never observed in practice with
`CustomEvent` from a child, but defensive.

### §2.5 Correlate by request_id, not by order

Multiple `callMcp()` calls can be in-flight concurrently on the
same plugin. Foundation filters incoming responses by
`detail.request_id`. The host MUST echo the exact request_id it
received — typo or transform breaks the correlation silently.

### §2.6 Propagate canonical error codes

When the bridge returns `{ok: false, code}`, pass the code
through unchanged. Plugin-author code does
`if (err.code === 'tool_not_found')` switches; rewriting the code
breaks that pattern. Drift #103 (canonical error-code propagation)
applies here.

Host-emitted codes for handler-layer failures:
- `no_active_plugin` — no plugin currently mounted (defensive)
- `invalid_request` — missing/empty `qualified_name` or `request_id`
- `forbidden_namespace` — namespace prefix didn't match active plugin
- `dispatch_failed` — IPC layer threw before reaching the bridge

The bridge layer's codes (`tool_not_found`, `insufficient_scope`,
`network_error`, etc.) pass through unchanged.

## §3 — Foundation `callMcp()` API

### §3.1 Import surface

```ts
import {
  callMcp,
  createCallMcpDispatcher,
  CallMcpError,
  CALL_MCP_DEFAULT_TIMEOUT_MS,
  PLUGIN_MCP_CALL_EVENT,
  PLUGIN_MCP_RESPONSE_EVENT,
  type CallMcpOptions,
  type PluginMcpActorClass,
  type PluginMcpCallDetail,
  type PluginMcpResponseDetail,
  type CallMcpDispatcher,
} from '@nexus-mindgarden/plugin-bridge-foundation/runtime'
```

All exports live under the `/runtime` subpath. The subpath is **browser-runtime
oriented** — it uses `crypto.randomUUID`, `CustomEvent`, `EventTarget`,
`HTMLElement`. Node 20+ provides all of these natively, so the module is
safely importable in node-vitest tests too.

### §3.2 Full signature (v0.6.1+ canonical)

```ts
function callMcp<T = unknown>(
  mount: HTMLElement,
  qualifiedName: string,
  args: unknown,
  options?: CallMcpOptions,
): Promise<T>

interface CallMcpOptions {
  actorClass?: 'user' | 'kiara'    // propagated as actor_class in request-detail
  timeoutMs?: number               // default 30000; set to 0 to disable
}
```

**Backward-compatibility:** the 3-arg form `callMcp(mount, name, args)`
produces **identical wire-output** to `callMcp(mount, name, args, {})`.
v0.6.0 consumers do not need to migrate; v0.6.1 adoption is opt-in per
call-site.

### §3.3 `CallMcpError` typed-class

```ts
class CallMcpError extends Error {
  readonly code: string
  // Inherits .name = 'CallMcpError', .message from Error
}
```

Designed for typed-catch idioms:

```ts
try {
  const result = await callMcp(mount, 'wiz-mind.character.list', {})
} catch (err) {
  if (err instanceof CallMcpError) {
    switch (err.code) {
      case 'tool_not_found':        // host-emitted (Drift #103)
      case 'insufficient_scope':    // host-emitted (Drift #103)
      case 'forbidden_namespace':   // host-emitted (cross-plugin-attack guard)
      case 'timeout':               // Foundation-emitted (options.timeoutMs elapsed)
      case 'crypto_unavailable':    // Foundation-emitted (insecure context)
        // …handle each code
    }
  }
}
```

The `.code` field is exposed as a separate property from `.message` so consumers
can switch on it without string-parsing. Drift #103 canonical error-codes
propagate verbatim from host through Foundation.

### §3.4 Timeout semantics (v0.6.1+)

- **Default:** `CALL_MCP_DEFAULT_TIMEOUT_MS = 30_000` (30 seconds)
- **On elapse:** rejects with `CallMcpError('timeout', "plugin:mcp-call '<qualifiedName>' timed out after <Xms> (request_id=<id>)")`
- **Opt-out:** `{ timeoutMs: 0 }` disables the timeout entirely (long-running stream pattern)
- **Cleanup invariant:** the timeout-handle is cleared on `resolve`, on `reject` from host-error, and on opt-out. No spurious late-firing rejections.

```ts
// Slow tool — extend timeout to 60s
await callMcp(mount, 'mind-canva.export.pdf', { layout_id }, { timeoutMs: 60_000 })

// Long-running stream — disable timeout
await callMcp(mount, 'wiz-mind.story.stream', { session_id }, { timeoutMs: 0 })
```

### §3.5 Curried dispatcher

```ts
const mcp = createCallMcpDispatcher(this)        // 'this' = custom-element root
await mcp('wiz-mind.session.start', { campaign_id })
await mcp('wiz-mind.scene.advance', { scene_id })
await mcp('mind-canva.brand_kit.get', { tenant_id }, { actorClass: 'kiara' })
```

Useful for components that issue multiple MCP-calls — avoids re-passing
`mount` everywhere. The 3rd `options` parameter (v0.6.1+) is forwarded
verbatim to the underlying `callMcp()` call.

### §3.6 Insecure-context behavior

Foundation rejects **before any event-dispatch** if `crypto.randomUUID` is
unavailable:

```ts
if (typeof globalThis.crypto === 'undefined' || typeof globalThis.crypto.randomUUID !== 'function') {
  return Promise.reject(new CallMcpError('crypto_unavailable', '...'))
}
```

This catches: Node < 14.17, browsers without secure-context (HTTP non-localhost),
older mobile WebViews, environments where `crypto` is removed by sandboxing.
The strict-rejection behavior is by design — silent fallback to `Math.random`
would leak request_id-collision-risk into security-sensitive flows.

## §4 — Plugin-Author Migration Recipe

_Contributed by `wiz-mind` (joint-smoke-first-mover). Based on Wiz-Mind's
`play.svelte` cut (commit [`9e6f8c5`](https://github.com/MrDewitt88/Wiz-Mind/commit/9e6f8c5))._

> Migration from ad-hoc CustomEvent dispatchers (pre-v0.6.0) to canonical
> Foundation `callMcp()` (v0.6.0+).

### §4.1 Before: ad-hoc CustomEvent + public-API write-back

```ts
// 1. Component dispatches per-plugin-named CustomEvents on the custom-element root
function dispatch(eventName: string, detail: Record<string, unknown>) {
  const el = document.querySelector('plugin-wiz-mind-play') ?? document.body
  el.dispatchEvent(new CustomEvent(eventName, { detail, bubbles: true, composed: true }))
}

async function loadCharacters() {
  dispatch('wiz-mind-list-characters', {})
  // wait for host to call setCharacters() via window.WizMind.setCharacters
}

async function handleStart() {
  dispatch('wiz-mind-session-start', { character_id, world_id })
  // wait for host to call setSessionStartResult()
}

// 2. Public-API surface for host to write results back via element-properties
export function setCharacters(list: CharacterSummary[]): void {
  characters = list
  charactersLoaded = true
  if (!selectedCharacter && list[0]) selectedCharacter = list[0].id
}

export function setSessionStartResult(r: { session?: { id: string }; error?: Error }): void {
  if (r.error) {
    startState = 'error'
    return
  }
  sessionId = r.session?.id
}
```

**Pre-v0.6.0 host-side glue (per-plugin event-listener wiring in mymind):**

```ts
// mymind ChatApp.svelte — separate handler per plugin × per event-name
playEl.addEventListener('wiz-mind-list-characters', async () => {
  const { characters } = await mcp.call('wiz-mind.character.list', {})
  playEl.setCharacters(characters)
})
playEl.addEventListener('wiz-mind-session-start', async (ev) => {
  try {
    const session = await mcp.call('wiz-mind.session.start', ev.detail)
    playEl.setSessionStartResult({ session })
  } catch (error) {
    playEl.setSessionStartResult({ error })
  }
})
// ... 3 more event-listeners per plugin × per tool
```

**Issues with this pattern:**

- Per-plugin glue in host = doesn't scale
- Race conditions between dispatch + write-back (no request_id correlation)
- Public-API surface (`setX`) bloats the component's contract
- Each plugin invents its own event-naming convention → drift

### §4.2 After: canonical `callMcp()` (Foundation v0.6.0+)

```ts
import { callMcp, CallMcpError } from '@nexus-mindgarden/plugin-bridge-foundation/runtime'

function getMount(): HTMLElement {
  return (document.querySelector('plugin-wiz-mind-play') as HTMLElement) ?? document.body
}

async function loadCharacters() {
  try {
    const { characters } = await callMcp<{ characters: CharacterSummary[] }>(
      getMount(),
      'wiz-mind.character.list',
      {},
    )
    // resolve sets local state directly, no host glue needed
    this.characters = characters
    if (!selectedCharacter && characters[0]) selectedCharacter = characters[0].id
  } catch (err) {
    if (err instanceof CallMcpError) {
      this.error = `${err.code}${err.message ? ': ' + err.message : ''}`
    } else {
      this.error = String(err)
    }
  }
}

async function handleStart() {
  try {
    const session = await callMcp<{ id: string }>(getMount(), 'wiz-mind.session.start', {
      character_id: selectedCharacter,
      world_id: selectedWorld,
    })
    sessionId = session.id // switches the view to scene-mode in-place
  } catch (err) {
    /* same error-handling shape */
  }
}

// no public-API export needed — component is self-contained
```

**Post-v0.6.0 host-side glue (ONE canonical handler, plugin-agnostic):**

```ts
// mymind ChatApp.svelte — ONE handler validates namespace + routes to bridge
mount.addEventListener('plugin:mcp-call', async (e) => {
  const { request_id, qualified_name, arguments: args, actor_class } = e.detail
  // Security: namespace MUST match the plugin's pluginId (cross-plugin-attack-guard)
  if (!qualified_name.startsWith(pluginId + '.')) {
    e.target.dispatchEvent(
      new CustomEvent('plugin:mcp-response', {
        detail: { request_id, ok: false, code: 'forbidden_namespace' },
        bubbles: true,
        composed: true,
      }),
    )
    return
  }
  const result = await api.pluginsDispatchTool({ qualifiedName: qualified_name, arguments: args })
  // RESPONSE dispatched on event.target (= the plugin's own custom-element)
  e.target.dispatchEvent(
    new CustomEvent('plugin:mcp-response', {
      detail: { request_id, ...result },
      bubbles: true,
      composed: true,
    }),
  )
})
```

### §4.3 Migration net-effect

|                    | Lines / files                      | Notes                                                                               |
| ------------------ | ---------------------------------- | ----------------------------------------------------------------------------------- |
| Removed            | -5 `setX` public-API methods       | `setCharacters`, `setWorlds`, `setSessionStartResult`, `setSceneResult`, `setError` |
| Removed            | -1 `dispatch(name, detail)` helper | replaced by `callMcp(mount, name, args)`                                            |
| Added              | +5 async-callsites                 | one per MCP-tool call                                                               |
| Bundle size        | +1.6 kb                            | callMcp runtime + CallMcpError handling                                             |
| Host code (mymind) | **-N event-listeners per plugin**  | from per-plugin glue to one canonical handler                                       |

**Net-effect:**

- cleaner state-flow (callMcp resolve writes local state directly)
- single failure-mode (`CallMcpError` with `code` field, Drift #103 compliant)
- no race-conditions (request_id correlates each call to its own response)
- plugin-agnostic host = zero per-plugin code in mymind for tool-routing

### §4.4 Migration step-by-step (~1 hour for 5 callsites)

1. `pnpm add @nexus-mindgarden/plugin-bridge-foundation@^0.6.0` (if not already on v0.6+)
2. Replace per-plugin event-names (`wiz-mind-list-characters`) with canonical tool-calls (`callMcp(mount, 'wiz-mind.character.list', {})`)
3. Remove `setX` public-API exports — `await callMcp()` returns the result directly
4. Replace `try { dispatch(...); }` blocks with `try { await callMcp(...); } catch (CallMcpError)` pattern
5. Update tests: mock `callMcp` (or use `MockedFunction`) instead of spying on `dispatchEvent`
6. Verify in browser DevTools: `[plugin:mcp-call] enter` + `resolved` logs (mymind host-side) appear for each call

## §5 — Failure-Mode Catalog — DOM-Bubble-Direction Anekdote

_Contributed by `wiz-mind` + `agent` (joint co-investigation, 2026-05-22).
A 2-hour debug session that surfaced a subtle interaction between host-side
event dispatch + plugin-side listener-attachment that **silently times out**
with no diagnostic. Documented here so others don't burn the same time._

### §5.0 Host-Internal Diagnostic Trace (Companion to §5.1)

_Contributed by `agent` (Luma) — host-internal perspective._

The §5.1 anekdote below describes the discovery from the plugin-author
perspective. From the host-internal side, the same bug had a different
set of signals that hosts implementing this protocol should know to
look for:

**What the host log shows when the bug is active:**

```
[plugin:mcp-call] enter request_id=<uuid> qualified_name=<tool> plugin=<id>
[plugin:mcp-call] resolved request_id=<uuid> dt=<ms> ok=true
```

The host correctly:
1. Caught the request via the ancestor listener
2. Validated the namespace
3. Routed to the bridge
4. Got `ok: true` back
5. Logged "resolved"

But the plugin's `callMcp()` Promise stayed pending until its 30s
timeout. The disconnect between host-side "resolved" and plugin-side
"timed out" is the diagnostic fingerprint: response IPC came back,
host emitted the CustomEvent, but the bundle never saw it.

**Structured log lines that surface this earlier:**

If a host emits both the host-side "resolved" log AND the renderer-
side "Foundation listener fired" log (latter requires Foundation
debug-mode), the gap between them isolates the dispatch-target
problem in <5 seconds rather than waiting for the 30s timeout.

Hosts implementing this protocol SHOULD log entry + resolve with
matching `request_id` (so debugging sessions can grep) AND consider
emitting a separate log when their response-dispatch's `EventTarget`
matches the original `event.target` (deviating = bug).

### §5.1 Symptom

Foundation `callMcp` dispatches `plugin:mcp-call` correctly, mymind's
`onMcpCall` IPC routing fires correctly, bridge tool-handler runs + returns
`{characters: [...]}` in ~3ms. Foundation's promise hangs.

30 seconds later:

```
CallMcpError: timeout: plugin:mcp-call 'wiz-mind.character.list' timed out
after 30000ms (request_id=abc-123-...)
```

No error logged on host-side. No error logged in bridge. Just a silent
timeout that looks identical to "tool implementation is broken."

### §5.2 Diagnosis path

| Step                                                                       | Verified? |
| -------------------------------------------------------------------------- | --------- |
| 1. `[plugin:mcp-call] enter` log fires on host                             | ✓         |
| 2. `pluginsDispatchTool` IPC roundtrip succeeds                            | ✓         |
| 3. Bridge tool-handler executes + returns `{ok: true, result}`             | ✓         |
| 4. `plugin:mcp-response` event is dispatched (somewhere)                   | ✓         |
| 5. Foundation's `mount.addEventListener('plugin:mcp-response', ...)` fires | ❌        |

Step 4 + 5 are the gap. Agent added a global `document.addEventListener('plugin:mcp-call', ...)` trap in DevTools and confirmed the event WAS being dispatched — just not where Foundation was listening.

### §5.3 Root cause

DOM tree at runtime:

```
pluginMountEl (div.plugin-mount)        ← mymind dispatched RESPONSE here
  └── <plugin-wiz-mind-play>             ← Foundation listens for response HERE
      └── (svelte template)              ← REQUEST dispatched here, bubbles up
```

**Request-path (works):**

- Foundation: `mount.dispatchEvent(plugin:mcp-call)` on the play-element
- Event bubbles UP to `pluginMountEl` (parent div)
- mymind's listener fires ✓

**Response-path (broken):**

- mymind dispatched `plugin:mcp-response` on **`pluginMountEl`** (parent div) ❌
- Event bubbles UP to body → html → document
- The play-element is a DESCENDANT, not an ancestor
- Foundation's listener (attached on play-element via `mount.addEventListener`) is **never triggered**

DOM event-bubble direction goes UP (child→parent). A response dispatched on
PARENT does not reach the descendant's listener — only window/document-level
listeners would catch it.

### §5.4 Fix

mymind dispatches the response on `event.target` (= the plugin's own
custom-element, where the request originated) instead of on `mount` (the
parent container).

```ts
// apps/mymind/src/renderer-main/ChatApp.svelte — onMcpCall handler
const responseTarget: EventTarget = event.target ?? mount // ← changed from just `mount`
responseTarget.dispatchEvent(
  new CustomEvent('plugin:mcp-response', {
    detail: { request_id, ...payload },
    bubbles: true,
    composed: true,
  }),
)
```

`event.target` is the element where the request was DISPATCHED FROM — i.e.
the plugin's own custom-element. The response now reaches Foundation's
listener via target-phase + bubbling.

**Fix commit:** Theseus-Agent
[`0aff0c5`](https://github.com/MrDewitt88/Theseus-Agent/commit/0aff0c5) on
branch `fix/mcp-response-target`. 1-line change. No regression risk.

### §5.5 Asymmetric symptom (red herring)

During diagnosis, `wiz-mind.character.list` appeared to "work" — it returned
an empty array and the UI rendered the "no characters yet" branch. We
assumed list resolved correctly while only create timed out.

**Reality:** list also timed out at 30s, but the timeout-thrown
`CallMcpError` was caught silently by code that also handled the
empty-result case (`charactersLoaded = true` in both paths). The user saw
"No characters yet — create your first below" which looked like a real
empty-list response.

**Lesson:** asymmetric symptoms in callMcp pipeline are often **the same
underlying bug** masked by lenient error-handling in one branch. Always
check whether the "working" calls actually fired their listeners.

### §5.6 Debug helpers

When troubleshooting silent callMcp timeouts, add to DevTools console
(BEFORE the suspected callMcp dispatch):

```js
window.addEventListener(
  'plugin:mcp-call',
  (e) => console.log('[WIN-trap] mcp-call', e.target, e.detail),
  true, // capture-phase
)
document.addEventListener(
  'plugin:mcp-call',
  (e) => console.log('[DOC-trap] mcp-call', e.target, e.detail),
  true,
)
window.addEventListener(
  'plugin:mcp-response',
  (e) => console.log('[WIN-trap] mcp-response', e.target, e.detail),
  true,
)
document.addEventListener(
  'plugin:mcp-response',
  (e) => console.log('[DOC-trap] mcp-response', e.target, e.detail),
  true,
)
```

If the request `[WIN-trap]` fires but the response `[WIN-trap]` doesn't —
the host isn't dispatching response. If response `[WIN-trap]` fires but
`mount.addEventListener` doesn't get called — bubble-direction issue.

### §5.7 Failure-mode catalog summary

| Failure                                           | Symptom                                                                   | Root-cause                                                           | Fix                                                                           |
| ------------------------------------------------- | ------------------------------------------------------------------------- | -------------------------------------------------------------------- | ----------------------------------------------------------------------------- |
| **DOM-bubble-direction mismatch**                 | silent 30s timeout, no logs anywhere except Foundation's own timeout      | host dispatches response on parent of plugin's mount-element         | host MUST use `event.target` from the request-event when dispatching response |
| **Cross-frame isolation** (untested, theoretical) | timeout, `composed:true` cannot cross iframe boundaries                   | plugin custom-element in iframe but host listener in parent document | use postMessage instead, OR mount plugin in same document                     |
| **Stale BUILD_ID cache**                          | callMcp call uses an old bundle's wire-shape after host wire-spec changes | browser cached `play-bundle.js` with `cache-control: immutable`      | wiz-mind bridge appends `?v=<BUILD_ID>` cache-buster on every bridge-restart  |

### §5.8 Foundation contract (post-fix)

> **Host MUST dispatch `plugin:mcp-response` on the same element-tree that
> dispatched the corresponding `plugin:mcp-call`** — specifically, on the
> `event.target` of the request-event (or any ancestor — descendants will
> not receive bubbled events).

Foundation v0.6.1+ adds `timeoutMs: 0` opt-out for streams, but the
underlying contract above is independent of timeout configuration. Any
host implementation must honor the bubble-direction constraint.

## §6 — Security Model

### §6.1 Foundation responsibilities (plug-tmpl)

**Foundation `callMcp()` enforces:**

- **`crypto.randomUUID` strict-rejection** — no silent fallback to weaker
  randomness sources. If the runtime is insecure-context (neither `https`
  nor `localhost`), the call rejects before any event-dispatch with
  `CallMcpError('crypto_unavailable')`. This prevents `request_id` collision
  attacks in environments where `Math.random()` is the only fallback.
- **`request_id` correlation strict-match** — listener filters incoming
  responses by exact `request_id` equality. A malicious bundle cannot
  trick Foundation into resolving a `callMcp` Promise with a response
  destined for a different in-flight call.
- **Listener cleanup invariant** — on resolve/reject/timeout, the
  event-listener is removed. Even with 1000 in-flight calls, Foundation
  registers one listener per call and removes it deterministically.
- **Timeout cleanup invariant** — on resolve/reject, the timeout-handle is
  cleared. No late-firing timeout can mutate state after the promise has
  settled.

**Foundation does NOT enforce:**

- **Cross-plugin namespace-guard** — Foundation cannot know which plugin is
  currently mounted; only the host has that context. The host MUST validate
  `qualified_name.startsWith(activePluginId + '.')`. See §6.2.
- **Token contents / scope-checks** — Foundation's `callMcp` is a transport
  helper. Auth-bearer-tokens never traverse the DOM event-stream; they live
  in the host's IPC layer between mymind and the plugin-bridge. The DOM is
  trusted only for plugin-author-to-host wire, not for token-passthrough.
- **Replay-protection on `request_id`** — Foundation generates unique IDs
  via `crypto.randomUUID` but does NOT cache emitted IDs to detect replay.
  A malicious bundle could replay an old `request_id` to confuse responses;
  this is in scope for host-side IPC-layer defense (next §6.2).

### §6.2 Host Responsibilities

_Contributed by `agent` (Luma)._

The host carries three security-critical responsibilities:

#### §6.2.1 Namespace enforcement is host-side

Foundation's `callMcp()` cannot enforce cross-plugin namespace
isolation — it has no knowledge of which plugin owns the active
DOM context. The host is the only component that knows the
mapping `mount → pluginId`. Therefore:

- The namespace-prefix check MUST happen host-side, before IPC
  dispatch
- A pure, unit-tested validator function isolates the rule from
  template code (Theseus mymind ships 7 unit-tests for
  `validateMcpCallNamespace`, including prefix-attack and
  multi-dot-segment cases)
- The check MUST reject the request with `forbidden_namespace`
  before any token resolution happens — leaking even the
  existence of a tool in another plugin's namespace is an
  information leak

```ts
// Reference: Theseus-Agent plugins-page-helpers.test.ts shipped with
// commit 003d37f — 7 namespace-validation tests including:
//   - prefix-attack: 'wiz-mindfoo.tool' must reject for pluginId='wiz-mind'
//   - cross-plugin: 'mind-canva.x' must reject for pluginId='wiz-mind'
//   - multi-dot accept: 'wiz-mind.scene.current' must pass
//   - null/empty rejection paths
```

#### §6.2.2 Tokens never enter the DOM

Whatever auth mechanism the host uses to talk to the plugin-bridge
(bearer token, Ed25519 signature, mTLS) — the secret material
stays in the host's main process. The renderer/UI/bundle layer
gets only:

- IPC channel handles (not the underlying secret)
- Plugin-bridge HTTP responses (already-authenticated)
- `qualified_name` strings (public surface)

If a bundle could reach a token via the DOM, the cross-plugin
attack from §6.3 would have a second vector (steal-and-replay
instead of namespace-spoofing). Keep tokens out of the renderer
process entirely.

#### §6.2.3 Audit-trail for failed validations

Forbidden-namespace attempts SHOULD be logged with structured
fields:
- `request_id` (for correlation if the bundle was confused, not malicious)
- `activePluginId` + `requestedNamespace` (the actual mismatch)
- `timestamp` (for rate-limiting + pattern detection)

A bundle making repeated `forbidden_namespace` attempts is either
broken or hostile. The audit log is what lets the host (or its
operator) tell the difference.

A repeated `request_id` collision (same id used twice within a
short window) is similarly logworthy — it suggests either a bundle
bug or a replay attack. Foundation's `callMcp()` generates fresh
UUIDs per call, so duplicate request_ids from a single bundle
should be ~zero in practice.

### §6.3 Cross-plugin attack scenarios (informational)

These attack-scenarios are guarded by §6.2 host-side enforcement and §6.1
Foundation-side enforcement combined:

**Scenario A: cross-plugin call attempt**

A `wiz-mind` plugin-bundle dispatches `plugin:mcp-call` with
`qualified_name: 'mind-canva.layout.create'`. The host MUST reject with
`code: 'forbidden_namespace'` because `mind-canva.` does not start with
`wiz-mind.`.

**Scenario B: namespace-prefix attack**

A `wiz-mind` plugin-bundle dispatches `qualified_name: 'wiz-mindfoo.tool'`,
hoping `startsWith('wiz-mind')` (without the dot) would let it through. The
host MUST reject — agent's mymind enforcement uses `startsWith(pluginId + '.')`
with the trailing dot specifically to block this.

**Scenario C: request_id collision**

Foundation's `crypto.randomUUID` guarantee makes this practically impossible
under normal use. The strict-rejection on insecure-context (§6.1) prevents
fallback to weak randomness. Hostside IPC-layer defense against deliberate
collision is in scope for §6.2.
